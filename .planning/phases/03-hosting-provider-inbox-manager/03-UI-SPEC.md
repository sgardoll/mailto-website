## Component Inventory

> All components are implemented as plain HTML + CSS + Vanilla JS. No third-party components.
> Reuse existing CSS classes where the pattern already exists. Add new classes only for Phase 3 additions.

### Provider Dropdown (HOST-01, HOST-06)

Element: `<select id="hosting-provider" name="hosting_provider">`

Styling: Use existing input border/radius tokens. Apply the same `border: 1px solid #d1d5db; border-radius: 6px; padding: 8px 12px; font-size: 16px; width: 100%; background-color: #ffffff; color: #111827;` rules already applied to text inputs. Add a dedicated CSS rule for `select` to match input appearance.

Focus state: `outline: 2px solid #2563eb; outline-offset: 2px; border-color: #2563eb;` — identical to existing input focus rule.

Options (in display order):
1. `value="siteground"` — "SiteGround"
2. `value="ssh_sftp"` — "Generic SSH/SFTP"
3. `value="netlify"` — "Netlify"
4. `value="vercel"` — "Vercel"
5. `value="github_pages"` — "GitHub Pages"

Default selected: first option ("SiteGround") — server pre-selects via `value` attribute on `<option selected>`.

Change event: JS listens to `change` on the `<select>`. On each change, it iterates all `.provider-fields` groups, adds `hidden` attribute to all, then removes `hidden` from the group matching `dataset.provider === select.value`. This mirrors the existing show/hide pattern in `wizard.js`.

No inline error needed on the dropdown itself — provider selection is always valid (one of 5 options is always chosen).

### Conditional Provider Field Groups (HOST-02–HOST-05, HOST-06)

Each provider's credential fields are wrapped in:

```html
<div class="provider-fields" data-provider="{value}" hidden>
  <!-- field groups for this provider -->
</div>
```

The active provider's group has `hidden` removed by JS on `change` and on initial page load. All other groups remain `hidden`.

CSS: `.provider-fields[hidden] { display: none; }` — use the same `.error[hidden]` pattern already in `wizard.css` line 121. Add this rule to `wizard.css`.

Field groups within each `.provider-fields` div use the standard `.field-group` structure (no new CSS needed).

**SiteGround fields** (`data-provider="siteground"`):
- `host` — `type="text"`, required, no default
- `port` — `type="number" min="1" max="65535" step="1" value="18765"`, required, default pre-filled: 18765
- `username` — `type="text"`, required
- `ssh_key_path` — `type="text"`, optional (see D-05 below)
- `password` — `type="password"` with Show/Hide toggle, optional (see D-05 below)
- `remote_base_path` — `type="text"`, required

**Generic SSH/SFTP fields** (`data-provider="ssh_sftp"`):
- `host` — `type="text"`, required
- `port` — `type="number" min="1" max="65535" step="1" value="22"`, required, default pre-filled: 22
- `username` — `type="text"`, required
- `ssh_key_path` — `type="text"`, optional (see D-05)
- `password` — `type="password"` with Show/Hide toggle, optional (see D-05)
- `remote_base_path` — `type="text"`, required

Note (D-05): For SSH-based providers, `ssh_key_path` and `password` are both always visible. Validation rule: at least one of the two must be non-empty. Both may not be blank. If both are blank on submit, show error on `ssh_key_path`: "Enter an SSH key path or a password — at least one is required."

**Netlify fields** (`data-provider="netlify"`):
- `netlify_api_token` — `type="password"` with Show/Hide toggle, required
- `netlify_site_id` — `type="text"`, required

**Vercel fields** (`data-provider="vercel"`):
- `vercel_api_token` — `type="password"` with Show/Hide toggle, required
- `vercel_project_id` — `type="text"`, required

**GitHub Pages fields** (`data-provider="github_pages"`):
- `gh_pages_branch` — `type="text" value="gh-pages"`, required, default pre-filled

### Inbox Row Manager (INBOX-01–INBOX-04)

Layout pattern: mirrors the Allowed Senders widget in structure. JS clones a `<template>` element (not DOM construction) to append new rows — cleaner than the sender-row createElement approach used in Phase 2.

Each inbox occupies a two-row layout within the 640px container. Row 1 holds the slug and email address (two columns, 50/50 split using `.field-row-split`). Row 2 holds site name, site URL, and base path (three columns, using a new `.field-row-thirds` class — see CSS additions below).

Container:

```html
<div id="inboxes-list">
  <!-- inbox rows appended here -->
</div>
<button type="button" id="add-inbox">Add inbox</button>
```

Single inbox row structure:

```html
<div class="inbox-row">
  <div class="inbox-row-header">
    <span class="inbox-row-label">Inbox {N}</span>
    <button type="button" class="remove-inbox" aria-label="Remove inbox {N}">Remove</button>
  </div>
  <div class="field-row-split">
    <div class="field-group">
      <label>Slug</label>
      <div class="field-row">
        <input type="text" class="inbox-slug" name="inbox_slug[]" ... />
      </div>
      <span class="error" hidden aria-live="polite"></span>
    </div>
    <div class="field-group">
      <label>Email address</label>
      <div class="field-row">
        <input type="email" class="inbox-email" name="inbox_email[]" ... />
      </div>
      <span class="error" hidden aria-live="polite"></span>
    </div>
  </div>
  <div class="field-row-thirds">
    <div class="field-group">
      <label>Site name</label>
      <div class="field-row">
        <input type="text" class="inbox-site-name" name="inbox_site_name[]" ... />
      </div>
      <span class="error" hidden aria-live="polite"></span>
    </div>
    <div class="field-group">
      <label>Site URL</label>
      <div class="field-row">
        <input type="url" class="inbox-site-url" name="inbox_site_url[]" ... />
      </div>
      <span class="error" hidden aria-live="polite"></span>
    </div>
    <div class="field-group">
      <label>Base path</label>
      <div class="field-row">
        <input type="text" class="inbox-base-path" name="inbox_base_path[]" value="/" ... />
      </div>
      <span class="error" hidden aria-live="polite"></span>
    </div>
  </div>
</div>
```

The `<template id="inbox-row-template">` element in the HTML contains one `.inbox-row` with all fields. JS clones it via `template.content.cloneNode(true)`, updates the label counter, appends to `#inboxes-list`, then attaches validation listeners.

### Add Inbox Button (INBOX-02)

Element: `<button type="button" id="add-inbox">`

Styling: identical to `#add-sender` — `font-size: 14px; color: #2563eb; background: none; border: none; padding: 4px 0; cursor: pointer; text-decoration: underline;`

Label: "Add inbox"

Placement: immediately below `#inboxes-list`, before the form actions block.

### Remove Inbox Button (INBOX-03)

Element: `<button type="button" class="remove-inbox" aria-label="Remove inbox {N}">`

Styling: identical to `.remove-sender` — `min-height: 44px; padding: 0 12px; border: 1px solid #d1d5db; border-radius: 6px; background-color: #ffffff; color: #dc2626; font-size: 14px; cursor: pointer; white-space: nowrap; flex-shrink: 0;`

Disabled state: `disabled` attribute set when only 1 inbox row exists. Styling: `color: #d1d5db; cursor: not-allowed;` — reuse `.remove-sender:disabled` rule. Add `.remove-inbox:disabled` with the same declaration.

JS: after every add or remove, call `updateRemoveButtons()` (same pattern as `updateRemoveButtons()` in the senders widget).

Placement: in `.inbox-row-header`, right-aligned via flexbox (`display: flex; justify-content: space-between; align-items: center; margin-bottom: 8px;`).

### Slug Uniqueness Validation (INBOX-04)

Trigger: `blur` event on any `.inbox-slug` input.

Algorithm: collect all `.inbox-slug` values into an array, lowercase and trim each. Find duplicates. For each `.inbox-slug` input whose value appears more than once in the array, show an inline error on that input's sibling `.error` span. For inputs with unique values, clear any existing duplicate error on the same span.

Error copy: "Slug must be unique across all inboxes"

Re-validation: on `blur` of any slug field, re-evaluate ALL slug fields for uniqueness (not just the one blurred). This ensures error clears when the user corrects a duplicate in another row.

Server-side enforcement: `/validate-form` with `step='inboxes'` also checks slug uniqueness and returns `{ok: false, errors: [{field: 'inbox_slug', index: N, message: '...'}]}` for each duplicate. JS maps server errors to the correct row's error span by index.

| sg-ssh-key-path / ssh-ssh-key-path | Conditional | At least one of key_path or password non-empty | "Enter an SSH key path or a password — at least one is required" |
| sg-password / ssh-password | Conditional | Same as above (error shown on key_path field) | — |
| sg-remote-base-path / ssh-remote-base-path | Yes | Non-empty | "Remote base path is required" |
| netlify-api-token | Yes | Non-empty | "API token is required" |

| gh-pages-branch | Yes | Non-empty | "Target branch is required" |

Touched-flag rule: identical to Phase 2 — validate on `blur` if `touched === true`; set `touched = true` on first `blur`. Re-validate on `input` once touched.

Conditional validation: only validate fields within the currently visible `.provider-fields` group. Fields in hidden groups are excluded from both client and server validation.

### Inboxes Step Field Validation

| Field | Required | Validation Rule | Error Copy |
|-------|----------|-----------------|------------|
| inbox_slug | Yes | Non-empty; `/^[a-z0-9-]+$/` pattern; unique across all rows | Non-empty: "Slug is required" / Pattern: "Slug may only contain lowercase letters, numbers, and hyphens" / Duplicate: "Slug must be unique across all inboxes" |
| inbox_email | Yes | Email validity (`input.validity.typeMismatch === false`) | "Enter a valid email address" |
| inbox_site_name | Yes | Non-empty | "Site name is required" |
| inbox_site_url | Yes | `new URL(value)` does not throw | "Enter a valid URL (e.g. https://example.com)" |

Minimum inbox count: 1. JS prevents removing the last row (disabled Remove button). Server also rejects `inboxes` step if the inboxes array is empty: error on list level — "At least one inbox is required."

## Accessibility Contract

Carried forward from Phase 2. Phase 3 additions:

- Provider `<select>` has `<label for="hosting-provider">` linked by `id`
- Each `.provider-fields` group has `aria-hidden` toggled alongside the `hidden` attribute: `group.hidden = true; group.setAttribute('aria-hidden', 'true')` when hiding; reverse when showing
- Each inbox row's inputs have `aria-describedby` referencing their sibling `.error` span ID
- Error spans within inbox rows use `aria-live="polite"`
- `aria-invalid="true"` set on slug inputs with active errors (both format and uniqueness)
- Remove button `aria-label` updated to "Remove inbox {N}" — JS updates on add/remove to keep counter accurate
- Minimum touch target 44px `min-height` on all Remove and add-row buttons
- Slug uniqueness re-validates all slug fields on each blur — all affected inputs get `aria-invalid` and visible error text simultaneously, not just the most recently touched one
- Focus order follows DOM order within each `.inbox-row`, then across rows in document order
